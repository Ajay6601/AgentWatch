# AgentWatch — Technical Design Document

## 1. What This Is

AgentWatch is a prototype of the classification engine that powers AI agent observability. It answers one question: **given a natural language description of a problem, can we automatically build a classifier that finds every occurrence of that problem across millions of agent traces?**

The answer is yes — by distilling a large LLM's judgment into a small, fast, per-issue classifier.

This document covers the architecture, design decisions, current results, and what a production system at Raindrop's scale would look like.

## 2. The Core Insight

AI agent observability has a cost problem:

| Approach | Accuracy | Cost at 100M traces/day | Latency |
|----------|----------|------------------------|---------|
| Run GPT-4 on every trace | ~100% | ~$9,600/day per issue | ~500ms/trace |
| Train a classifier on labeled data | 90-95% | ~$370/day per issue | ~15ms/trace |
| No classification (manual review) | — | Free but impossible | — |

The classifier approach is 96% cheaper but requires labeled training data. The traditional way to get labels is to have humans annotate thousands of examples — slow, expensive, doesn't scale to arbitrary issue types.

**The distillation approach bridges this gap:** use the expensive LLM to label a small sample (~100 traces), train the cheap classifier on those labels, then discard the LLM and run the classifier on everything. Each new issue type gets its own bespoke classifier, trained in minutes from just a natural language description.

This is what Raindrop calls "bespoke few-shot classifiers" — and it's what this prototype implements.

## 3. Pipeline Architecture

```
┌─────────────────┐     ┌──────────────┐     ┌───────────────┐     ┌──────────────────┐
│  Natural lang.  │     │  LLM Judge   │     │  Few-Shot      │     │  Bulk            │
│  issue desc.    │ ──► │  labels 100  │ ──► │  Classifier    │ ──► │  Classification  │
│  from engineer  │     │  traces      │     │  trains on     │     │  runs on all     │
│                 │     │  ($0.01)     │     │  those labels  │     │  traces          │
└─────────────────┘     └──────────────┘     └───────────────┘     └──────────────────┘
                              │                      │                       │
                        Strong teacher          Fast student           Production output
                        (expensive, slow)      (cheap, fast)         (frequency, users,
                                                                      flagged traces)
```

### 3.1 Trace Generation

For this prototype, traces are generated by running GPT-4o-mini with different system prompts that induce specific failure modes:

- **Normal**: Agent uses tools, gives detailed specific answers
- **Lazy**: Agent skips tools, gives generic deflections ("check the docs")
- **Forgetting**: Agent asks for information the user already provided
- **Hallucination**: Agent confidently invents features, policies, or pricing
- **Tool loop**: Agent calls the same tool 3-5 times without progress

This produces real LLM-generated traces with natural variance — the agent doesn't respond the same way twice, tool outputs differ, token counts vary. The classifier learns behavioral patterns, not templates.

**Cost**: ~$0.15 for 500 traces via GPT-4o-mini.

### 3.2 LLM-as-Judge Labeling

The judge receives an issue description and a trace, and returns a binary label with confidence:

```json
{"label": true, "confidence": 0.92, "reasoning": "Agent gave generic 'check the docs' response without searching KB"}
```

**Key design decisions:**
- Temperature 0.05 for consistent labeling
- Confidence threshold ≥0.80 to filter out uncertain labels
- Stratified sampling to ensure diverse traces in the labeled set
- Async batched calls (5 concurrent) for throughput
- Structured evaluation criteria in the prompt (tool usage, response specificity, actionability)

**Observed behavior**: The judge works well for clearly defined issues like laziness (14-17% positive rate, close to ground truth). For ambiguous issues like "tool loop," the positive rate inflates (50%+) because the boundary between "thorough search" and "loop" is subjective. Production systems would need per-issue prompt tuning.

### 3.3 Few-Shot Classifier Training

Three approaches, each with different tradeoffs:

**SetFit** (best accuracy): Fine-tunes a sentence-transformer using contrastive learning on text pairs, then fits a classification head. With 50-80 labeled examples, it consistently achieves F1 > 0.95. Training takes ~18 minutes on CPU, ~3 minutes on GPU. Model size: ~22MB.

**Embedding + Logistic Regression** (fastest training): Encodes all traces with a frozen sentence-transformer, fits sklearn LogisticRegression with balanced class weights. Trains in 2-3 seconds. F1 = 0.85-1.00 depending on the eval split. Most practical for rapid iteration.

**Prototype Network** (fewest examples needed): Computes class centroids in embedding space, classifies by cosine/euclidean distance to nearest centroid. Works with as few as 5 examples. F1 = 0.80. Useful as a fast baseline or when the LLM judge returns very few positive labels.

**Class balancing**: Minority class oversampling (up to 4x) prevents the classifier from defaulting to the majority class when the judge returns imbalanced labels.

### 3.4 Bulk Classification

The trained classifier runs on all traces with a confidence threshold (≥0.70) to reduce false positives. Output per issue:
- Number of flagged traces and percentage (frequency)
- Number of affected users
- List of flagged trace IDs with confidence scores
- Link to each trace's full span execution

### 3.5 Unknown Unknowns Discovery

HDBSCAN clusters all trace embeddings without requiring a predefined number of clusters. Each cluster is auto-labeled by sending representative traces to GPT-4o-mini for summarization.

**Discovered patterns from 500 traces:**
- "Repeated Incorrect Response" (20 traces, 4.6%)
- "Repetitive Request for Customer ID" (19 traces, 4.3%)
- "Inconsistent response truncation" (16 traces, 3.6%)
- "Customer Lookup Failure" (14 traces, 3.2%)

These are issues nobody searched for — the system found them on its own.

## 5. Design Decisions

### Why SetFit over LoRA/PEFT fine-tuning?

We're training binary classifiers on 50-100 examples, not adapting a foundation model to a new domain. SetFit is purpose-built for few-shot text classification — it uses contrastive learning to reshape the embedding space, then fits a tiny classification head. LoRA fine-tuning would be like using a sledgehammer to hang a picture frame.

### Why HDBSCAN over K-Means?

The "unknown unknowns" problem requires clustering without knowing how many failure types exist. HDBSCAN is density-based — it finds clusters of varying shapes and marks sparse points as noise. K-Means forces you to pick k upfront, which defeats the purpose of discovery.

### Why OpenTelemetry-compatible traces?

Raindrop's SDK integrates via OTEL trace exporters (confirmed in their docs for Vercel AI SDK, Node.js, and Cloudflare Workers). Using the same trace_id/span_id/parent_span_id format means this pipeline could ingest real production traces with zero format conversion.

### Why three classifier approaches?

Different issues and data conditions favor different approaches:
- SetFit: best when you have 30+ labeled examples and can afford 18 min training
- Embedding+LR: best for rapid iteration (2s training) or when you need interpretability
- Prototype: best when the judge returns very few positive examples (5-10)

In production, you'd auto-select based on the label distribution.

## 6. Limitations

**1. Synthetic traces.** Generated via prompted LLM calls, not captured from real production agents. Real failure patterns are messier — the classification pipeline is designed for that messiness, but the current results reflect cleaner-than-real data.

**2. Single-tenant.** No model isolation between customers. Production needs per-tenant classifier namespaces with independent training and versioning.

**3. Batch-only classification.** Traces are classified after ingestion, not at ingest time. Production needs streaming classification via a message queue.

**4. CPU-only inference at ~15-28ms/trace.** On GPU, sentence-transformer inference drops to ~1-3ms/trace, which would show 200-500x speedup over the LLM judge instead of the current 18-20x.

**5. Judge prompt sensitivity.** The LLM judge works well for clearly defined issues (laziness) but over-labels ambiguous issues (tool loops). Production would need per-issue prompt tuning or a calibration step.

## 7. Production Architecture

### What I'd Build at Raindrop Scale

**Per-tenant model management:**
```
tenant_001/
├── issues/
│   ├── laziness/
│   │   ├── v1/ (setfit model, deployed 2024-03-01)
│   │   ├── v2/ (retrained with active learning labels)
│   │   └── active → v2
│   └── hallucination/
│       └── v1/
tenant_002/
└── issues/
    └── custom_issue/
        └── v1/
```

**Streaming classification pipeline:**
```
Trace ingestion (OTEL SDK)
    → Kafka topic (raw traces)
    → Classification worker pool
        → Run all active classifiers for this tenant
        → Write predictions to Tinybird (analytics)
        → If any issue frequency spikes → Slack alert
    → FAISS index (for Deep Search similarity queries)
```

**Active learning loop:**
```
1. Classifier runs on new traces
2. Traces with confidence 0.4-0.6 (uncertain zone) get queued
3. LLM judge labels the uncertain traces
4. Labels added to training set → retrain
5. A/B test new classifier vs old → promote if better
```

This creates a flywheel: the classifier improves over time by focusing expensive LLM calls on the examples it's least sure about. Cost grows sublinearly with trace volume.

**GPU optimization:** Multiple tenants share the same sentence-transformer encoder. Encode once, run multiple classification heads. The marginal cost of an additional issue classifier per tenant is nearly zero — it's just a logistic regression head or a SetFit classification layer on top of shared embeddings.

## 8. Metrics That Matter

For an AI observability product, adoption depends on:

| Metric | Target | Current |
|--------|--------|---------|
| Time to first insight | < 30s | ~19 min (CPU SetFit training dominates) |
| Classifier agreement with LLM judge | > 90% | 93-96% (SetFit) |
| Cost per 1M traces classified | < $5 | $3.63-3.79 |
| False positive rate | < 10% | ~8% (with 0.70 confidence threshold) |
| Unknown patterns discovered | > 0 | 35 clusters from 500 traces |

The biggest gap is time-to-first-insight. On GPU, SetFit training drops from 18 minutes to ~3 minutes. With embedding+LR as a fast default (2s training, F1=0.86) and SetFit as a background refinement, the user sees results in under 30 seconds.
